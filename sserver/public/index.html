Performance Research

This simple HTML page can be used to navigate to a number of performance experiments carried out in JavaScript, WASM, ASM and C. You will need to run this page using the sserver tool to ensure that all files are served properly. If anyone would like to make this page look pretty, you are welcome to do so!

The purpose of these experiments is to observe the computational and runtime performance between C, ASM.js, WebAssembly and hand written JavaScript. For initial runtime observations, hardware requirements and software versions used visit the Wiki.

Delaunay Triangulator

The following experiment uses an original Delaunay Triangulator code written in 1993 by Geoff Leach. Small modifications to the timer functions were required to be able to compile and run using ASM.js and WASM. The large majority of the code remains unchanged from the original source.

Delaunay Triangulator is an algorithm that takes a set of 2D random points and triangulates them in such a way that all triangles are non-intersecting. It involves a large number of number comparison operators and is memory bandwidth intensive.

Matrix Computation

The following experiment uses a ported three.js open source code. Small modifications have been made to ensure that the JavaScript source and the C source matches as closely as possible. This source matching is required to ensure the benchmark is as fair as possible.

three.js is an open source javascript graphics library. This experiment uses a number of 4x4 Matrix, Vector and Quaternion operations to try and emulate a similar scenario that a typical graphics application would perform on a regular basis. These operations are typically compute heavy with minimal CPU branching operations.

Random and Sort

This experiment uses C's qsort and JavaScripts built in TimSort and/or MergeSort to compare the performance between different browsers against ASM.JS and WebAssembly. The reason why the code-bases are not the same is because JavaScript's sorting algorithms vary between vendors. For an apples to apples comparison, please refer to other experiments.

The code uses the same source of XorShift64 Random Number Generator to generate a set of random numbers and sorts them with qsort (C) or JavaScript's built in sort.

Repeated Function Calls

This experiment repeatedly calls a WASM function which performs a computation on a random 4x4 matrix and returns its determinant as a single 32 bit float value. The purpose of this experiment is to find any significant overheads with function calls to WASM.

The code uses the same source of XorShift64 Random Number Generator to generate a set of random numbers and uses the same 4x4 matrix code to compute it's determinent.
